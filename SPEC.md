# Flop Go DB Engine Spec

## Goal

Flop is a code-first embedded database and app runtime for realtime web apps.

This spec describes the `go1` engine behavior and public semantics, not every internal optimization. It is intended as a rebuild target for another language/runtime.

## Core Model

- Database contains named tables.
- Table schema is declared in code.
- First schema field is the primary key.
- Rows are schemaless at storage boundary but validated against compiled schema on write.
- Queries are not SQL-based.
- Public app API is built from:
  - table CRUD
  - typed `view` handlers
  - typed `reducer` handlers

## Data Types

Supported logical field kinds:

- `string`
- `number`
- `integer`
- `boolean`
- `timestamp`
- `json`
- `enum`
- `vector`
- `set`
- `roles`
- `bcrypt`
- `ref`
- `refMulti`
- `fileSingle`
- `fileMulti`

Field capabilities:

- required
- default value
- autogenerated value
- unique constraint
- secondary index
- full-text index
- field read/write access policy
- reference metadata (`refTable`, `refField`)
- cascade archive marker on refs

## Storage Layout

Each live table uses:

- `{table}.flop` for row pages
- `{table}.wal` for write-ahead log
- `{table}.idx` for persisted primary index
- `{table}.midx` for mapped primary index

Each archived table uses:

- `{table}.archive.flop` for archived row records
- `{table}.archive.idx`
- `{table}.archive.midx`

Shared/system data:

- `_meta.flop` stores schema history and table metadata
- `_files/...` stores live file assets
- `_archive_files/...` stores archived file assets

## Page Format

Live rows are stored in fixed-size slotted pages.

- Page header stores page metadata and LSN watermark.
- Slot directory grows forward.
- Row payload grows backward.
- Delete marks slot tombstoned.
- Update may happen in place if new row fits, otherwise row relocates.

Archive rows are stored in the same page-file style, but payload is an archived-record envelope instead of a normal row.

## Row Encoding

Live row payload contains:

- schema version
- ordered field values encoded in binary

Archived row payload contains JSON-encoded metadata envelope:

- `archiveId`
- `originalPk`
- `deletedAtUnixMs`
- `deletedBy`
- `cascadeGroupId`
- `cascadeRootTable`
- `cascadeRootPk`
- `cascadeDepth`
- `rowData` (original encoded live row bytes)

## Schema and Migration Semantics

- Schema is compiled from code on startup.
- Schema changes are versioned.
- Old rows are lazily migrated on read.
- Migration chain is resolved from stored schema history plus declared migration rules.

## Index Model

### Primary Index

- One primary index per table.
- Maps primary key string to page/slot pointer.
- Persisted separately from row pages.
- Can be rebuilt from table file.

### Secondary Indexes

Supported index classes:

- unique hash index
- non-unique multi index
- full-text index

Properties:

- secondary indexes are derived state
- may be rebuilt on startup
- uniqueness is enforced on write

## Durability Model

Writes use WAL-before-checkpoint semantics.

### WAL

WAL records include:

- begin
- insert
- update
- delete
- archive insert
- archive delete
- commit

Properties:

- WAL records have checksum
- WAL records carry transaction id
- WAL replay restores committed changes after crash
- group commit is allowed

### Checkpoint

Checkpoint flushes:

- dirty row pages
- archive pages
- primary index
- archive index
- persisted secondary indexes
- checkpoint manifest

After successful checkpoint:

- WAL checkpoint LSN advances
- WAL may be truncated

## Consistency Model

### Single Operations

Single insert/update/delete/archive/restore is atomic with respect to durability:

- WAL intent is built
- in-memory/page changes apply
- WAL batch commits
- checkpoint happens later

### Public Transactions

Public reducer transaction API exists:

- `Transaction(ctx, func(tx *Tx) ...)`

Transaction guarantees:

- buffered WAL commit across touched tables
- all writes commit together or fail together
- rollback hooks restore live/archive state for typed CRUD and archive/restore operations

Current limitation:

- reads inside a transaction are not full MVCC snapshot reads over uncommitted writes
- transaction is currently a write atomicity boundary, not a full serializable read model

## Write Semantics

### Insert

- apply defaults/autogenerated values
- validate field types
- validate unique constraints
- serialize row
- append WAL
- place row in page
- update primary and secondary indexes

### Update

- load existing row
- merge updates
- validate schema
- validate uniqueness for changed indexed fields
- write in place if row still fits, else relocate
- update indexes as needed

### Delete

Hard delete semantics still exist at low level:

- remove row from live table
- remove indexes
- remove live files

Public app-facing lifecycle should prefer archive instead.

### Archive

Archive operation:

1. read current live row bytes
2. create archived envelope
3. write archived envelope to archive store
4. remove live row from live store
5. move row files from `_files` to `_archive_files`
6. persist both archive insert and live delete in one WAL transaction

### Restore

Restore operation:

1. read archived record
2. verify live PK is free
3. reinsert original live row bytes
4. remove archived record
5. move files back to `_files`
6. persist both live insert and archive delete in one WAL transaction

## Cascade Archive / Restore

Cascade is declared on reference fields.

Current cascade action:

- `CascadeArchive`

Behavior:

- archiving a root row archives all rows that reference it through cascade-marked refs
- all archived rows in one cascade share `cascadeGroupId`
- root metadata is stored on every archived child
- restore of a root archive restores the whole cascade group in depth order

Expected restore ordering:

- parent/root first
- deeper descendants after ancestors

## File Semantics

File fields store metadata in row values and bytes on disk.

Live delete/archive behavior:

- hard delete removes row file directory
- archive moves row file directory into archive storage
- restore moves archived files back

## Access Control

There are two access layers.

### Endpoint Access

Views and reducers can be:

- public
- authenticated
- roles-based

`superadmin` bypasses route role checks.

### Table / Field Access

Table access policies can control:

- read
- insert
- update
- delete

Field access policies can control:

- read
- write

Current model is callback-based, not compiled predicate-based.

Read filtering semantics:

- a row may be hidden entirely
- individual fields may be removed from returned row

## Query Model

No general SQL engine exists.

Queries are composed from:

- `Get(pk)`
- `Scan(limit, offset)`
- unique index lookup
- non-unique index lookup
- composite index lookup
- full-text search

Views and reducers are typed functions registered in code and exposed automatically as HTTP endpoints.

## Validation Model

### Input

- reducer JSON input is strictly decoded
- unknown JSON fields are rejected
- view query params are strictly decoded
- unknown query params are rejected

### Row Writes

- writes validate field kinds and constraints
- bcrypt fields are hashed on insert/update as appropriate
- unique constraints are enforced before commit

### Output

- typed output exists at code level
- there is no separate runtime output validator today

## Realtime Model

- in-process pubsub emits change events per table
- SSE/admin refresh flows subscribe to change events
- events include table, op, row id, optional row data

## Admin Surface

Admin panel supports:

- listing live tables
- listing archive tables
- browsing live rows
- browsing archived rows
- creating/updating/deleting live rows
- restoring archived rows
- backup/restore endpoints
- materialized table refresh
- analytics and observability screens

Archive admin rows expose extra metadata fields:

- `_archiveId`
- `_originalPk`
- `_deletedAt`
- `_deletedBy`
- `_cascadeGroupId`
- `_cascadeDepth`

## Public Typed API

Typed tables support:

- `Insert(scope, row)`
- `Get(scope, id)`
- `Update(scope, id, ...updates)`
- `Delete(scope, id)`
- `Scan(scope, limit, offset)`
- `Count(scope)`

Valid scopes:

- `*ViewCtx`
- `*ReducerCtx`
- `*DBAccessor`
- `*Database`
- `*Tx`

Transaction API:

- `Transaction(ctx *ReducerCtx, fn func(*Tx) (T, error))`

## Rebuild Requirements

Any rewrite should preserve these externally visible behaviors:

- schema-first tables
- WAL durability with replay
- persisted/rebuildable indexes
- typed views/reducers as public app model
- strict reducer/view input validation
- archive instead of destructive delete as preferred lifecycle
- cascade archive/restore groups
- file archive/restore movement
- admin archive browsing and restore
- public typed transaction API with atomic commit/rollback

## Allowed Internal Changes

A rewrite may change:

- page size
- exact binary layout
- index data structures
- memory mapping strategy
- concurrency primitives
- checkpoint policy
- query planner internals

As long as the visible semantics above remain true.
